# P12n

P12n is the active crypto trading project with a machine-learning focus.

The current center of gravity is not generic quantitative research. It is an
applied system for finding, validating, and trading low-SNR return signals in a
crypto setting. The project pulls from broader quant methods, but its pages
should stay anchored to current modeling, execution, and infrastructure work.

## Map

- [N-Linear Returns Models](p12n/n-linear-returns-models.md): the current
  structured model family for lagged multi-asset return forecasting.
- [Temporal Returns Experiments](p12n/temporal-returns-experiments.md):
  lagged return signal studies, short-lag predictiveness, round-minute effects,
  and temporal basis discovery.
- [Sequence-Model Analogies](p12n/sequence-model-analogies.md): RNN,
  linear-attention, test-time regression, and adaptive-filter lenses that inform
  model design.
- [Feature Transforms And BST](p12n/feature-transforms-and-bst.md): bin
  smoother trees, product bins, transform grammars, out-of-fold derived
  predictions, and leakage control.
- [Execution And Policy](p12n/execution-and-policy.md): BPTE, thresholds,
  forecast-to-policy translation, fill-rate targets, and realized utility.
- [Experiment Infrastructure](p12n/experiment-infrastructure.md): `duct`,
  caching, config management, reports, and model comparison machinery.

## Current Shape

P12n has two interlocking research loops.

The first loop is return forecasting. Recent notes focus on temporal returns
experiments, n-linear model iterations, lag bases, cross-asset structure,
in-context-learning-like local regression ideas, and ways to make nonlinear or
sequence-model-like ideas compatible with closed-form or alternating
least-squares fitting.

The second loop is trading policy. Forecast quality is not enough by itself.
The execution side has to translate forecasts into thresholds, position targets,
fill-rate behavior, and risk-aware utility under realistic market mechanics.
This inherits lessons from [Obelisk](obelisk.md), especially the separation
between entry signal quality and realized strategy value.

The supporting engineering loop is about making experiments cheap and
auditable: streaming feature transforms, lazy recomputation, model/statistic
comparison, and explicit separation between forecasting, execution, and
threshold training.
