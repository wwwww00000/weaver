# Problem Solving And Skill Acquisition

Whetstone treats problem solving as a trainable composite skill. The point is
not to become generally "smarter" in the abstract. The point is to develop a
small set of operators, representations, drills, and review loops that improve
real attempts at coding, math, research, writing, and building.

The source note is blunt: exercise the problem-solving muscle, develop
metacognitive skill, reflect on the reasoning chain and techniques used, and
regularly attempt Codeforces-style problems. This page turns that intention into
a practice model.

## Competency Map

Problem solving is not one ability. Useful competencies include:

- analytical decomposition: break a problem into parts;
- symbolic manipulation: operate reliably on formal representations;
- working memory: hold the active state without losing the thread;
- pattern recognition: notice familiar structure;
- representation choice: draw, verbalize, formalize, simulate, or simplify;
- insight incubation: spend time with a problem until a better frame emerges;
- procedural fluency: apply known methods without re-deriving everything;
- forecasting: look several moves ahead;
- creative search: generate alternate approaches;
- metacognitive control: notice when the current strategy is wrong;
- communication: explain the solution and expose gaps.

Different people can be strong in different competencies. A Whetstone practice
should not assume one universal solver style. It should identify the missing
competency for a given class of failure.

## Problem-Solving Loop

Most loop models collapse into the same structure:

1. Observe or define the problem.
2. Orient: identify constraints, knowns, unknowns, and current representation.
3. Choose a strategy.
4. Execute for a bounded interval.
5. Check the result against evidence.
6. Reflect, update the frame, and repeat.

This is OODA, PDCA, build-measure-learn, and metacognitive problem solving in
Whetstone language. The important move is not naming the loop. It is keeping the
loop short enough that mistakes update the next attempt.

## Operators

When the `stuck` tag fires, the next move should be concrete. Useful operators:

- restate the problem without inherited wording;
- list constraints and degrees of freedom;
- solve a smaller case;
- test an extreme case;
- search for an invariant;
- draw the state space;
- make a table of examples and counterexamples;
- reverse the direction of reasoning;
- replace variables with numbers;
- separate syntax from semantics;
- ask what would make the problem easy;
- explain the current partial model out loud;
- compare with a known problem family;
- stop and write a restart note.

The operator library should be practiced directly. Codeforces, project Euler,
small proof exercises, debugging tasks, and toy research questions all work if
the review names the operator that succeeded or failed.

## Visual-Spatial Reasoning

Visualization is not only for art. It can be used as a problem-solving
representation.

Useful forms:

- mind maps for branching associations;
- concept maps for labeled relationships;
- storyboards for processes, user flows, or arguments;
- mental rotation for geometry, mechanics, code paths, or scene blocking;
- object models where variables become movable pieces with properties;
- spatial layouts where constraints occupy positions;
- diagrams of data flow, control flow, dependencies, and feedback loops;
- simulations where the system is operated in the mind before building.

The Tesla-style move is design rehearsal: build the thing internally, operate
it, notice faults, revise, then externalize. For Whetstone, the practice should
always end with a checkable artifact: sketch, diagram, pseudocode, proof
outline, test case, or implementation note.

## Skill Acquisition Loop

Skill acquisition uses the same controller loop, but the object of control is
practice quality.

1. Pick a target skill or subskill.
2. Decompose it into trainable components.
3. Choose one bottleneck component.
4. Design a short drill with feedback.
5. Execute with attention to error signatures.
6. Adjust the drill or return to the project.
7. Log the insight in a reusable form.

This is deliberate practice, reflective practice, self-monitoring, and kaizen
under one roof. The useful Whetstone move is microanalysis: after a failure,
name the specific fault rather than judging the entire attempt.

Examples:

- drawing: visualize the line before drawing it;
- music: hear the note before playing it;
- math: predict the shape of a proof before reading;
- coding: trace the invariant before typing;
- writing: state the paragraph's job before polishing sentences.

## Projects Versus Drills

Project-based learning supplies relevance, motivation, and whole-system
context. Drills supply focused repetition on bottlenecks.

Use projects when:

- the domain is applied or open-ended;
- motivation depends on a live problem;
- the missing pieces are not yet known;
- integration matters more than component fluency.

Use drills when:

- a component repeatedly bottlenecks the project;
- speed or accuracy matters;
- a foundational skill is weak;
- the feedback signal can be made tight.

The default Whetstone pattern is project first, drill the bottleneck, then
return to the project. Revelation's math-learning page uses the same principle:
go depth-first enough to stay motivated, but fill prerequisites when they become
real blockers.

## Previsualization

Previsualization is a useful bridge between cognition and motor or symbolic
skill. Before acting, build a short internal preview of the intended move:

- the line before drawing;
- the note before playing;
- the sentence before speaking;
- the proof skeleton before reading details;
- the code path before editing;
- the experiment outcome before running it.

The preview is not a fantasy of success. It is a prediction. After the action,
compare predicted and actual outcome. That makes mental imagery trainable
rather than decorative.

## Practice Design

A Whetstone practice block should be small enough to repeat:

- warm up with one easy item;
- pick one operator or bottleneck;
- attempt a hard-but-contained task;
- stop at a natural checkpoint;
- write a two-minute postmortem;
- extract one drill, operator, or representation for next time.

Mind-gym-style circuits can be useful if they remain tied to transfer. Working
memory drills, mental math, visualization, divergent-thinking games, and puzzle
sprints should be judged by whether they improve real tasks, not by whether the
drill score goes up.

## Transfer Checks

Cognitive training is vulnerable to fake progress. Track transfer explicitly:

- Does the drill improve a real target task?
- Does the operator appear unprompted during hard work?
- Does the postmortem identify a different failure than last week?
- Does recovery from stuck states get faster?
- Does the practice improve output quality, not only mood?

If a drill only produces local improvement, keep it as a warmup or game. Promote
it to Whetstone practice only when it changes real problem-solving behavior.
